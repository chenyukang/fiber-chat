# fiber-chat：用 Fiber 做去中心化社交的消息层

最近做了一个小 demo，叫 [fiber-chat](https://github.com/chenyukang/fiber-chat)。

它表面上是一个网页聊天应用，但底层并不是走传统的中心化消息服务，而是把消息写进 Fiber Network 的 `send_payment.custom_records`，让一条消息跟随一笔 Layer 2 payment 在网络中传递。

我做这个 demo，关注点其实不在“聊天界面”本身，而在另外一个问题：

**Fiber 这样的支付网络，能不能顺手承担去中心化社交里的消息传输层？**

做完这版 demo 之后，我觉得这个方向是成立的。原因很简单：去中心化社交最难的几个问题，Fiber 恰好能同时碰到。

## 去中心化社交到底难在哪

如果只说“把消息从 A 发到 B”，这件事不难。真正难的是下面这几件事：

- 谁来帮你转发消息
- 为什么别人愿意帮你转发
- 怎么控制垃圾消息和滥发
- 中间节点参与转发时，能不能看见内容

很多所谓去中心化社交，最后还是绕回了中心化服务器。原因并不神秘，因为消息分发、节点激励、隐私保护这几件事，一旦放在一起做，就没那么简单。

而 `fiber-chat` 想验证的，就是能不能借 Fiber 把这几件事放进同一个模型里。

## fiber-chat 的核心思路

在这个 demo 里，发一条消息，不是“调用聊天接口”，而是：

1. 把消息编码成 JSON
2. 写进 `send_payment.custom_records`
3. 发起一笔很小的 keysend payment
4. 让这笔 payment 沿着 Fiber 网络路由到目标节点

也就是说，这个 demo 不是在 Fiber 上“模拟聊天”，而是真的把消息塞进了 payment session。

从产品表面看，这是一条聊天消息。
从底层实现看，这其实是一笔带自定义数据的 Layer 2 payment。

## 为什么这件事很像去中心化社交

### 1. 发消息是有成本的

在 `fiber-chat` 里，发送者需要支付一笔极小额的 CKB 才能把消息发出去。

这个金额可以很小，小到不影响正常聊天体验，但它有一个非常重要的作用：
**消息不是零成本的。**

这对去中心化社交很关键。

在中心化社交里，平台可以通过风控、审核、封号去处理垃圾消息；但在去中心化网络里，如果发送消息完全没有成本，spam 和滥发会很快变成系统性问题。

而在 Fiber 这个模型下，消息天然带着一张很轻的“邮票”。
平时聊天几乎无感，但如果有人想大规模灌垃圾消息，成本会立刻堆起来。

所以它至少提供了一种比“全靠治理”更底层的约束方式。

### 2. 中间节点转发消息可以获得收益

去中心化社交还有一个常见问题：
**为什么中间节点要帮你转发？**

很多设计默认节点会主动参与网络维护，但现实里节点要开机、联网、维护状态，还要承担带宽和资源消耗。如果没有激励，网络很难长期稳定。

Fiber 天然不一样，因为它本来就是一个带 fee 的路由网络。

在 `fiber-chat` 里，一条消息不是广播出去的，而是跟随一笔 payment 沿着已有通道路由。只要消息经过中间 hop，中间节点就可以因为转发获得一小部分 fee。

这个模型很自然：
- 发送者支付一笔很小的 CKB
- 中间节点通过转发获得 fee
- 接收者收到消息

这就让“消息分发”第一次有了一个更像样的经济闭环。

从去中心化社交的角度看，这件事很重要，因为它意味着消息传播不只是技术上能跑起来，经济上也有持续运转的可能。

### 3. 中间节点参与路由，但看不到消息内容

如果只有“付费转发”，那它还只是一个有手续费的中继网络。
真正让我觉得这个方向有意思的，是另外一点：

**中间节点负责转发，但它们看不到聊天内容。**

这是因为消息的承载实际发生在 Fiber 的 onion packet 路由模型中。对于中间 hop 来说，它知道自己该往哪里转发，也知道这笔 payment 经过了自己，但它并不能直接读取聊天内容。

这点特别重要。在很多系统里，中间层虽然只是“转发者”，但实际上天然掌握了内容访问权。而在这个模型里，中间节点可以参与网络、获得 fee，却不天然拥有查看消息内容的能力。

这就很接近去中心化社交真正想要的东西了：

- 有分发
- 有激励
- 有隐私

如果说得更直白一点，中间节点在这个系统里更像“邮路”，而不是“编辑部”。

## 这个 demo 是怎么实现的

当前的 `fiber-chat` 主要分成三层。

### 前端

前端是一个聊天界面，支持：

- 按 node 查看聊天页
- 会话列表和未读状态
- 左右消息气泡
- 消息对应的 Fiber route、fee、payment hash、hop 信息展示

从体验上看，它像一个普通 IM；只是每条消息背后都对应着一次真实的 Fiber payment。

### Rust 后端

后端是整个 demo 的协调层，负责：

- 调用 Fiber JSON-RPC
- 自动 prepare 网络
- 把消息编码进 `custom_records`
- 发起 payment
- 轮询各节点 payment 记录，重建聊天时间线

这里有一个现实取舍：Fiber 当前的公开 RPC 更偏 payment/session 视角，并没有一个现成的“消息收件箱”接口。所以这个 demo 采用的是观察者式聚合方案，由后端从 payment 列表里筛出聊天记录，再交给前端展示。

### 本地 Fiber 网络

为了方便演示，项目里内置了一套本地 bundle：

- 一个本地 CKB dev chain
- 三个 Fiber 节点：`node1`、`node2`、`node3`

启动之后会自动：

- 给节点分配 CKB
- 启动本地 Fiber 节点
- 建立 channel
- 补双向 liquidity

所以这个项目虽然现在主要用于本地 demo，但底层机制是完整走通的，不是假的“前端模拟”。

如果你想本地跑一跑这个 demo，可以执行：

```bash
docker pull chenyukang/fiber-chat:latest
docker run --rm -p 3000:3000 chenyukang/fiber-chat:latest
```
或者通过源码参照 [README](https://github.com/chenyukang/fiber-chat?tab=readme-ov-file#quick-start) 里的说明来启动运行。


## 结语

`fiber-chat` 它不是在做一个完整社交产品，而是在测试一种可能的消息层。
做的事情其实很朴素：

**让一条聊天消息，跟着一笔极小额的 CKB payment 一起在 Fiber 网络中传播。**

但正因为采用了这种方式，这个 demo 同时具备了几件很难得的特性：

- 发送消息有成本
- 中间节点有激励
- 中间节点不能直接看到内容
- 消息传输建立在真实的 Layer 2 路由网络上

这几件事放在一起，我觉得已经很接近去中心化社交真正重要的部分了。

不是把中心化社交简单搬到链上，
而是从一开始就把消息、路由、激励和隐私放进同一个设计里去考虑。

如果以后真要出现更像样的去中心化社交基础设施，我觉得这条路值得继续往下走。
